# AI 駆動開発の方法論 (AI-Driven Development Methodology)

生成 AI は非決定論的(=同じ指示でも毎回出力が変わりうる)。そのまま任せると、仕様からのブレ・秘密の露出・未テストコードの混入が起きる。本書は、この性質を前提に **個人〜小規模チームで品質を保って開発を進めるための方法論** ── 実プロジェクトでの運用を通じて確立した手順を、任意のプロジェクトに適用できる形で公開する。

> **⚠️ 本書の賞味期限について**
> 本書は 2026 年 5 月時点の AI エージェント(Claude Code 等)を前提に書かれている。モデルとツールの進化は速く、具体的な設定・コマンド・「どこまで AI に委ねられるか」の線引きは今後変わりうる。
> ただし本書は **変わりにくい原則**(SoT・責務分解・多段レビュー・トレードオフ選定)と **変わりやすい実装例**(Claude Code の設定等)を分けて記述している。実装例が古びた場合は、原則を各時点の最新ツールに読み替えて適用してほしい。

**一言でいうと**: 「人間がレビューできる小さな単位に分け、AI に複数案を作らせ、機械的チェックと多段レビューで安全に絞り込む」。

## 背景 ── なぜこの文書を書いたか

AI に実装を任せ始めた頃、まず戸惑ったのは「同じ指示でも出力が毎回変わる」ことだった。細かく手順を固定すれば思い通りには動く。だが AI の推論が活きず、設計と食い違っても気づけない。かといって丸ごと委ねれば、仕様からのブレや未テストコードの混入が待っている。どちらに倒しても、品質は守れなかった。

行き着いた答えは、任せ方の工夫ではなく**役割の分離**だった──「判断は人間、実装は AI、品質は仕組みで担保」。判断基準を文書(SoT)に固定して AI には How だけを委ね、事故は注意力ではなくガードレールで止める。このフローで 1 つのアプリケーションを作り切った。本書はその試行錯誤の記録。同じ戸惑いの中にいる誰かの回り道を、少しでも短くできれば嬉しい。

## 対象読者

- AI コーディング支援(Claude Code 等)を使っているが、生成コードの品質担保に不安がある人
- 「AI に任せる範囲」と「人間が判断する範囲」の線引きに迷っている人
- 個人開発〜3 名程度の、レビュー体制が薄い環境の人

**前提知識**: Git / GitHub の基本操作(ブランチ、PR、Issue)。本書の例は Claude Code だが、原則はツール非依存(→ 適用のヒント)。

## 全体像と導入手順

流れは 4 ステップ。各ステップの詳細は対応章へ、手を動かす際は対応テンプレートへ。

| ステップ            | やること                                                             | 詳細   | テンプレート       |
| ------------------- | -------------------------------------------------------------------- | ------ | ------------------ |
| 1. 設計             | AI と壁打ちしながら技術選定を 1 つずつ確定し、設計書にまとめる       | 2 章   | ―                  |
| 2. 計画             | 設計書を「責務ごと」の小さな作業票(GitHub Issue)に分割する         | 3 章   | 付録 A             |
| 3. ガードレール     | 秘密の漏洩・危険な操作を、AI の注意力ではなく仕組みで機械的に止める  | 4 章   | 付録 B             |
| 4. 実装             | Issue 1 件ごとに AI へ 3 案を同時生成させ、多段レビューで最良案を選ぶ | 5・6 章 | 付録 C・D          |

> 用語: **Issue**(=作業を 1 件ずつ管理する票)/**責務**(=そのコードが担当する役割。例: 画面表示、データ保存)/**MVP**(=必要最小限で動く最初の版)/**ブランチ・マージ**(=作業用の枝で開発し、本流へ取り込むこと)。

## 原則: SoT ── すべての判断の基準を 1 か所に置く

> **背骨は SoT(Single Source of Truth=唯一の正解とする文書)。** あらゆる判断はここに照合して決める。
>
> - **要件の最終根拠**: 要件文書(requirements)
> - **設計判断の正準**: 設計書(design doc)
>
> プロンプトの「制約」・コードレビューの合否・候補案の選定まで、判断は必ず SoT に照合する。AI には **How(どう作るか)を委ね、What(何を作るか)と判断基準は SoT に置く** ── 非決定論的な生成でも仕様からブレさせないための原則。

## 1. 基本方針 ── 論点をトレース可能にする

- **責務単位への分解**: 人間がレビュー可能な「論点(責務)単位」で GitHub Issue へブレークダウンする。1 Issue = 1 責務とし、PR(=Pull Request、変更をまとめてレビューに出す単位)でレビューしやすい粒度を保つ。
- **発散と収束の使い分け**: アイデア出し(ブレスト)はスコープを限定せず広げる。実装フェーズは責務単位に収束させ、論点を 1 つずつ閉じていく。
- **高推論モデル前提のプロンプト設計**: 最新モデルの推論性能をフル活用するため、手順を細かく指定しない(過剰な手順指定はアンチパターン=避けるべき型)。プロンプトは原則 **目的 / 制約(SoT へのポインタ)/ 完了条件** の 3 点のみを与え、How は AI に委ねる。

  **補足: プロンプトの対比**(同じ「お気に入り機能の実装」を頼む場合)

  - **アンチパターン**(How=手順・クラス名・例外まで人間が固定)

    ```
    お気に入り API を実装して。まず favorites.controller.ts を作り @Controller('favorites') を付け、
    POST /favorites と DELETE /favorites/:id を生やす。:id は ParseIntPipe で受け、service.addFavorite を呼び、
    repository.save で保存し、重複時は ConflictException を投げ、DTO は @IsInt() を付けて……
    ```

    → 手順を固定するとモデルの推論を活かせず、設計書と食い違っても気づきにくく手戻りが増える。

  - **良いパターン**(目的 / 制約 / 完了条件の 3 点だけ渡す)

    ```
    目的: お気に入り機能(docs/issues/10-favorites-api.md)の実装。
    制約: 要件文書と設計書。MVP フェーズ。
    完了条件: 当該 Issue に記載済み。
    ```

    → What(何を)と判断の拠り所(SoT)だけ渡し、How は AI に委ねる。認可境界や ID 正規化などの正準は SoT 側が規定済みなので、細かく書かなくてもブレない。

## 2. 設計 ── ブレスト → 判断確定 → 設計書化

設計は AI との対話で **「ブレスト → 設計判断の確定 → 設計書化」** の順に進める。

1. 要件文書を読み込ませ、要件と評価軸を整理する。
2. 分岐点になる技術判断(backend / frontend スタック、DB、認証方式、キャッシュ有無)を AI が選択肢とトレードオフ(=一長一短の比較)付きで提示し、人間が **1 つずつ確定**する(後戻りを削減)。
3. 確定事項から AI が設計書ドラフトを生成し、人間がレビュー・承認する。

**有効なプロンプト技法**

- 役割を固定する(「あなたはシニアのテックリードです」)と、選定理由・拡張性・運用観点を含む回答が得られる。
- 選択肢とトレードオフを要求する(「2〜3 案を比較し、推奨案と理由を述べて」)と意思決定材料が揃う。
- 評価軸を明示する(「使い捨ての試作(PoC)でなく保守性・拡張性を重視」)と設計の厚みが評価軸に沿う。

**この段階の検証**

- 選定理由を人間が一つずつ吟味し、代替案との比較が成立しているか確認する。
- 設計書末尾に **要件トレーサビリティ表**(=要件と設計の対応表)を作り、要件文書の全要件と突合する。
- AI が暗黙に置いた前提(DB にデータを残さない、CSRF=なりすまし送信攻撃は対象外 等)を設計書に明記させ、第三者が判断できるようにする。

## 3. 計画 ── 設計書を Issue へ分解する

設計書を**実装単位の GitHub Issue** へ分解する(テンプレートは付録 A)。

- **粒度の基準**: 責務分離・依存関係・実装順序・完了条件・検証観点が読み取れること。1 Issue に複数責務を詰めず、PR レビューしやすい単位にする。
- **各 Issue の必須項目**: 背景 / 目的 / 対応内容 / 設計方針 / 完了条件 / やらないこと / 検証観点 / 依存 Issue。
- **特に重視する設計観点**:
  - 変更されやすい部分を**閉じ込める**設計になっているか。
  - **UI / hooks / API / service / repository / DB**(=画面 / 画面状態のロジック / 外部公開の窓口 / 業務ロジック / データ保存の出入口 / データベース)の責務境界が明確か。
- Issue 間で依存関係と実装順序を明示する。

## 4. ガードレール ── 危険を注意力でなく仕組みで止める

生成 AI は放置すると秘密の露出・破壊的操作・未テストコードの混入を起こしうる。AI の善意や記憶に頼らず、**ハーネス(=AI を動かす実行基盤)側で fail-safe(=問題を検知したら処理を進めず止める)に止める**(設定例は付録 B)。

- **テスト駆動(TDD) / e2e**(=テストを先に書いて実装を導く進め方 / 画面から DB まで通しで動かす検証): 実装したら都度テストを追加する。実行ツールは例として backend が Jest、frontend が Vitest。
- **エージェントのフック機構**(Claude Code では hooks=特定操作の前後に処理を自動実行する仕組み): ソースファイル編集時に lint(=書き方の自動チェック)と typecheck(=型の整合性チェック)を自動実行し、失敗したら止める。
- **権限制御**(Claude Code では `.claude/settings.json`):
  - 破壊的コマンド(ファイルを全削除する `rm -rf`、履歴を強制上書きする force push 等)を deny(=禁止)。
  - `.env` / `.env.*`(実シークレット)の Read を deny し、鍵の隠蔽をハーネス側でも担保する。
  - 日常の開発コマンド(lint / test 等)は allow(=許可)。
  - 個人差分は `.claude/settings.local.json`(非コミット)に分離する。

## 5. 実装フロー(Issue 1 件あたり)── 並列生成と多段レビュー

### A. 並列実装

Git の **worktree**(=同じリポジトリを複数の作業フォルダに分けて並行作業できる機能)を使い、**同一スコープを 3 案並列**に実装させ、最良を採用する。非決定論的な生成のブレを、複数案の比較選定で吸収する狙い。

各候補について、以下の多段レビューを通す。

| #   | ステップ                   | 目的                             | 制約 / インプット                                                                     | 完了条件                                     |
| --- | -------------------------- | -------------------------------- | ------------------------------------------------------------------------------------ | -------------------------------------------- |
| 1   | 初期プロンプトで実装       | 当該 Issue スコープの実装        | SoT 2 文書(要件 / 設計書)、MVP フェーズ                                             | Issue 記載の完了条件(実態は TDD が駆動)    |
| 2   | 説明文書を生成             | 短時間で実装を直感的に理解       | トップダウン・論理が追える・明確簡潔・網羅的・専門用語は括弧補足                       | 高度な経験のない一般エンジニアでも理解できる |
| 3   | 別系統 AI で差分レビュー   | 別系統 AI でクロスチェック       | コード差分                                                                            | 指摘を確認・反映                             |
| 4   | PR 作成(二層構成)        | 人間と AI 双方が追える PR にする | 上=平易な要約(なぜ/何を)、下=技術詳細(固有名は原文維持)、概念用語は括弧で一言補足 | ―                                            |
| 5   | セキュリティと多角レビュー | 抜けの検出                       | エージェント付属のセキュリティレビュー・多角的レビューコマンド                         | 指摘があれば対応、なければ次へ               |

> PR の概念用語補足の例: 「fail-fast(=起動時に問題を検知してすぐ止める)」のように括弧で一言だけ添え、固有名(クラス名・コマンド名等)は原文のまま触らない。人がさっと読め、AI・エンジニアが正確にたどれる。

### B. 候補選定(トレードオフ表)

a / b / c の実装を**トレードオフ表**で比較し、最良を採用する(フォーマットは付録 C)。判断基準は **SoT(要件文書・設計書)への適合度**。一般的なソフトウェアエンジニアでも表を見て比較判断できる粒度で提示させる。

### C. main マージ

選定した候補を main へマージする。問題があれば対応してから取り込む。

## 6. レビュー・検証 ── 何重で品質を守るか

- **人間レビュー**: AI 生成コードは小さい単位で確認する。セキュリティに関わる箇所(認証・Cookie・別ドメインからのアクセス許可を制御する CORS)は AI 任せにせず人間が重点レビューする。
- **多段の自動レビュー**: 別系統 AI の差分レビュー → エージェント付属のセキュリティ・多角的レビュー、の多層。
- **並列生成とトレードオフ選定**: 同一スコープ 3 案を SoT 適合度で比較し最良を採用する。
- **テスト**: 実装の都度テストを追加する(unit + e2e)。
- **設計妥当性・要件網羅**: 選定理由の吟味と、要件トレーサビリティ表での全要件突合。

### A. Issue テンプレート

```markdown
## 背景

<!-- なぜこの作業が必要か。設計書のどの部分に対応するか -->

## 目的

<!-- この Issue が完了すると何ができるようになるか(1〜2 行) -->

## 対応内容

<!-- 実装する内容の概要 -->

## 設計方針

<!-- 準拠する設計書の章、責務の置き場所(UI / hooks / API / service / repository / DB) -->

## 完了条件

- [ ] <!-- 検証可能な形で書く -->

## やらないこと

<!-- この Issue のスコープ外を明示(スコープの膨張を防ぐ) -->

## 検証観点

<!-- テストで確認すること。正常系 / 異常系 -->

## 依存 Issue

<!-- 先に完了している必要がある Issue -->
```

### B. ガードレール設定例(Claude Code `.claude/settings.json`)

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Bash(rm -rf *)",
      "Bash(git push --force*)"
    ],
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Bash(npm run build)",
      "Bash(git status)",
      "Bash(git diff *)"
    ]
  }
}
```

- ルールは `ツール名(パターン)` の形式。deny が最優先で評価される。
- lint / typecheck を編集のたびに自動実行するにはフック機構(hooks)を併用する。設定方法・最新の構文は [公式ドキュメント](https://code.claude.com/docs/en/settings) を参照。
- 個人環境の差分は `.claude/settings.local.json`(非コミット)へ。

### C. トレードオフ表フォーマット(候補選定用)

```markdown
| 観点                       | 候補 a | 候補 b | 候補 c |
| -------------------------- | ------ | ------ | ------ |
| 要件への適合               |        |        |        |
| 設計書への適合(責務境界) |        |        |        |
| テストの充実度             |        |        |        |
| 可読性・保守性             |        |        |        |
| 懸念点                     |        |        |        |

**採用**: 候補 _ / **理由**: (SoT 適合度を根拠に 1〜2 行)
```

### D. プロンプト雛形

**Issue 実装の初期プロンプト**(細かく指定せず、目的 / 制約 / 完了条件のみ)

```
目的: docs/issues/<issue-file>.md の Issue スコープの実装。
制約: 要件文書と設計書。MVP フェーズ。
完了条件: 当該 Issue に記載済み。
```

**実装後の説明文書を生成**(理解補助)

```
目的: 実装内容を、時間のない中で直感的に理解するのを手伝ってほしい。
制約: トップダウン。論理の流れが追いやすい。明確かつ簡潔。網羅的。専門用語は補足して。
完了条件: 高度な言語・フレームワーク・ライブラリ経験がない一般的なエンジニアでも理解できる。
```

## 適用のヒント

- **小さく始める**: いきなり全部を導入せず、まずガードレール(4 章)+ プロンプト雛形(付録 D)だけでも効果がある。並列 3 案生成は慣れてから。
- **並列数はコストと相談**: 3 案が基本だが、単純な Issue は 1 案 + 多段レビューで十分。判断が割れそうな Issue にこそ並列生成を使う。
- **他のエージェントでも使える**: 本書の例は Claude Code だが、SoT 原則・責務分解・多段レビュー・トレードオフ選定はツール非依存。フック / 権限制御は各ツールの同等機能に読み替える。
